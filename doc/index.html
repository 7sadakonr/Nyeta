Nyeta Project

การพัฒนาระบบผู้ช่วยจำแนกวัตถุภายในครัวเรือนแบบโต้ตอบเรียลไทม์ผ่านเว็บแอปพลิเคชัน
สำหรับผู้พิการทางสายตาโดยประยุกต์ใช้ปัญญาประดิษฐ์เชิงสร้างสรรค์

DEVELOPMENT OF AN INTERACTIVE REAL-TIME HOUSEHOLD OBJECT CLASSIFICATION ASSISTANT SYSTEM VIA WEB APPLICATION FOR VISUALLY IMPAIRED PERSONS USING GENERATIVE ARTIFICIAL INTELLIGENCE

🎯 บทนำและวัตถุประสงค์ของโครงงาน

ปัญหา (Pain Point)

ผู้พิการทางสายตาเผชิญความยากลำบากในการดำเนินชีวิตประจำวันภายในบ้าน (เช่น การแยกยา หรือของใช้) แอปพลิเคชันส่วนใหญ่เป็น Native App ที่ติดตั้งยาก ไม่รองรับภาษาไทยอย่างสมบูรณ์ และ AI มีข้อจำกัดเรื่อง ความแม่นยำในที่แสงน้อยจนอาจนำไปสู่อันตรายได้

แนวคิดการแก้ปัญหา (Solution)

พัฒนาระบบแบบผสมผสาน (Hybrid Architecture) ที่ใช้ AI นำทางมือแบบเรียลไทม์ ร่วมกับ Cloud LMM (Llama 4 Vision) และอุดช่องโหว่ความผิดพลาดของ AI ด้วย ระบบอาสาสมัคร (Human-in-the-Loop) ผ่านเว็บแอปพลิเคชันก้าวหน้า (PWA) ที่รองรับ Screen Reader 100%

วัตถุประสงค์ของโครงงาน

  • 1. เพื่อพัฒนาระบบผู้ช่วยอัจฉริยะสำหรับจำแนกและระบุตำแหน่งสิ่งของสำหรับผู้พิการทางสายตา ในรูปแบบ PWA
  • 2. เพื่อประยุกต์ใช้เทคโนโลยี AI ร่วมกับระบบเครือข่ายอาสาสมัคร (Volunteer Network System)
  • 3. เพื่อศึกษาและประเมินประสิทธิภาพของระบบต้นแบบ (ความแม่นยำของ AI และระยะเวลาตอบสนอง)

🔍 เจาะลึก: อัลกอริทึมทำงานอย่างไร? (คลิกเพื่ออ่านเพิ่ม)
  • 1. สแกนวัตถุเรียลไทม์ (useObjectDetector): รัน requestAnimationFrame โยนภาพเข้าโมเดล COCO-SSD (TF.js) ตลอดเวลา เพื่อหากรอบวัตถุ (Bounding Box)
  • 2. นำทางด้วยเสียง (Spatial Guidance): นำพิกัดวัตถุมาคำนวณเทียบกับจุดกึ่งกลางจอ (Center Zone 20%) หากวัตถุเอียงไปด้านใด จะมีเสียงพูดสั่งให้ "เลื่อนซ้าย/ขวา" พร้อมระบบหน่วงเวลา (Throttle) ไม่ให้พูดซ้ำรัวเกินไป
  • 3. ดักฟังเสียง: เมื่อผู้ใช้แตะหน้าจอเพื่อถามคำถาม ระบบจะเรียก SpeechRecognition (Web Speech API) แปลงเสียงพูดเป็นข้อความ
  • 4. สร้าง Base64 (useAiAssistant): ระบบแคปภาพวิดีโอ ณ วินาทีนั้นลงบน Canvas แปลงเป็น Base64 แล้วแพ็กคู่กับคำถาม ส่งผ่าน HTTP Request ไปหาคลาวด์ Groq Vision API
  • 5. รับผลลัพธ์: เมื่อ AI ตอบกลับมา ระบบจะโยนข้อความเข้า API Text-to-Speech เพื่ออ่านคำตอบให้ฟังทันที

โหมดสแกนธนบัตร (Currency Scanner)

Continuous Sampling + Vision AI

📸
สุ่มดึงภาพ (Frame Sampling) ดึงภาพนิ่งจากกล้องเป็นจังหวะเพื่อประหยัด Data
⬇️
🧠
คัดแยกธนบัตร (Vision AI) ให้ AI ค้นหาและจำแนกมูลค่าธนบัตรไทย
⬇️
📳
ยืนยันความแม่นยำ (Verify) เมื่อค่าตรงกันหลายเฟรม ระบบจะสั่นเตือน+บอกจำนวนเงิน
🔍 เจาะลึก: อัลกอริทึมตรวจสอบธนบัตร (คลิกเพื่ออ่านเพิ่ม)
  • 1. ตั้งเวลาแอบถ่าย (Interval): รัน setInterval สั่งจับภาพนิ่งเงียบๆ ทุก 4 วินาที เพื่อไม่ให้เปลืองเน็ต (หากมี Error จะใช้สมการยืดเวลาออกไป)
  • 2. จำแนก (Classify): ส่งภาพไปคลาวด์ โดยตั้ง Prompt บังคับให้ AI ตอบกลับมาเป็น JSON สั้นๆ เท่านั้น เช่น {"value": 100}
  • 3. กรองความแม่นยำ (State Machine):
    - รอบที่ 1: AI ตอบ 100 -> ระบบเก็บค่าไว้ใน stableDetectionRef แล้วเตือนว่า "กำลังยืนยัน" (ยังไม่สั่น)
    - รอบที่ 2: ถ่ายใหม่ AI ตอบ 100 อีก -> ตัวนับครบ 2 ระบบจึงจะยอมรับผล สั่งมอเตอร์มือถือให้สั่น (Haptic) และอ่านออกเสียง
  • 4. รีเซ็ต (Reset): หากหมุนกล้องไปทางอื่นจน AI หาไม่เจอติดกัน 2 ครั้ง ระบบจะล้างค่าทั้งหมดทิ้งเพื่อเริ่มสแกนใหม่

โหมดอ่านเอกสาร (Document Reader)

Edge Detection + Cloud OCR

📐
ตรวจจับขอบ (Local Edge Detect) หาพิกัด 4 มุมกระดาษแบบเรียลไทม์ พร้อมเสียงนำทาง
⬇️
📸
จับภาพ & ดึงข้อความ (Cloud OCR) เมื่อกล้องตรง ระบบจับภาพออโต้แล้วส่งทำ OCR บนคลาวด์
⬇️
🔊
อ่านออกเสียง (Continuous TTS) อ่านข้อความทั้งหมดให้ผู้ใช้ฟังยาวต่อเนื่อง
🔍 เจาะลึก: อัลกอริทึมอ่านเอกสาร (คลิกเพื่ออ่านเพิ่ม)
  • 1. ตั้งเวลาวิเคราะห์ (Interval): รัน setInterval ทุก 0.5 วินาที เพื่อเช็คเฟรมภาพจากกล้องตลอดเวลา
  • 2. หาพิกัด 4 มุม (Local): โยนภาพนิ่งเข้าไลบรารี scanic (WASM) หารูปทรงสี่เหลี่ยม จากนั้นใช้สมการคณิตศาสตร์ Distance เทียบความกว้าง/ความสูง เพื่อหาว่าหน้ากระดาษเอียง (Tilt/Roll) หรือตกขอบหรือไม่
  • 3. เสียงนำทาง (Throttled TTS): นำค่าความเอียงที่คำนวณได้แปลงเป็นคำสั่ง (เช่น "เงยกล้องขึ้น", "เลื่อนขวา") โดยมีระบบหน่วงเวลาไม่ให้พูดซ้ำติดกันเกินไป
  • 4. ถ่ายภาพอัตโนมัติ (Auto-Capture): เมื่อผู้ใช้จัดกล้องตรง (Aligned) ติดต่อกันครบ 3 เฟรม ระบบจะสั่งจับภาพเองโดยไม่ต้องให้คนตาบอดคลำหาปุ่มถ่าย
  • 5. ทำ OCR (Cloud): ส่งภาพไปหาคลาวด์ ดึงข้อความออกมาทั้งหมด และโยนเข้า Text-to-Speech เพื่ออ่านให้ฟังยาวรวดเดียว

💡 หลักการทำงานของ TURN Server

ปกติแล้ว WebRTC จะพยายามเชื่อมต่อวิดีโอแบบ Peer-to-Peer (P2P) ระหว่าง 2 อุปกรณ์โดยตรง แต่หากฝั่งใดฝั่งหนึ่งใช้อินเทอร์เน็ตที่มี Symmetric NAT หรือ Strict Firewall (เช่น เน็ตมือถือ 4G/5G บางค่าย หรือเน็ตองค์กร) การเชื่อมต่อตรงจะถูกบล็อกทันที

TURN Server จึงเข้ามาทำหน้าที่เป็น Relay Server (เซิร์ฟเวอร์ทางผ่าน) เมื่อเชื่อม P2P ไม่สำเร็จ อุปกรณ์ทั้งสองจะส่งแพ็กเก็ตวิดีโอไปที่ TURN Server แทน จากนั้นเซิร์ฟเวอร์จะทำหน้าที่ Forward ข้อมูลข้ามเครือข่ายไปให้อีกฝั่ง เป็นการการันตีว่าวิดีโอคอลจะเชื่อมต่อได้ 100%

🔍 เจาะลึก: อัลกอริทึมการเชื่อมต่อสาย (คลิกเพื่ออ่านเพิ่ม)
  • 1. แจ้งเตือนแบบกลุ่ม (Broadcast): เมื่อคนตาบอดกดโทร ระบบ (useBlindHelp) จะสร้าง Call ID ยิง Broadcast เข้า Channel กลาง ทำให้มือถืออาสาสมัครทุกคนที่ออนไลน์อยู่ ดังขึ้นมาพร้อมกัน
  • 2. กรองการรับสาย (Race Condition): อาสาสมัครคนไหนที่กดปุ่มรับสายเป็นคนแรก จะเป็นผู้ชนะ (Claimed) ระบบจะยิงคำสั่งกลับไปยกเลิกสายของคนอื่นๆ ทันที
  • 3. เริ่มจับคู่ (Signaling): คนตาบอดจะสร้างข้อเสนอเทคนิค SDP Offer แล้วแพ็กส่งข้ามอินเทอร์เน็ตผ่าน Pusher ไปให้ฝั่งอาสาสมัคร 1 คนนั้น
  • 4. หาเส้นทาง (ICE/TURN): ทั้งสองเครื่องจะแลก IP กัน (ICE Candidates) หากเชื่อมแบบ Direct P2P ไม่ทะลุ ระบบจะสลับไปพาข้อมูลวิ่งผ่าน TURN Server เป็นทางเบี่ยง
  • 5. ตัดสายอัตโนมัติ (Timeout): หากโทรไปแล้วไม่มีอาสาสมัครคนไหนกดรับสายเลยภายใน 40 วินาที ระบบฝั่งคนตาบอดจะตัดสายทิ้งอัตโนมัติ (Ring Timeout)

Hardware & Dev Constraints

ฝั่งผู้ใช้ (Client): ต้องมี กล้อง, ไมโครโฟน, ลำโพง และมอเตอร์สั่น (Haptic) เพื่อประสบการณ์การใช้งานที่สมบูรณ์

Environment Variables: ต้องตั้งค่าไฟล์ .env.local โดยใส่ API Key ของ Groq และ Pusher (App ID, Key, Secret) เพื่อให้ระบบ AI และระบบคอลอาสาสมัครทำงานได้

TURN Server (Optional): หากนำไปใช้บนเซิร์ฟเวอร์จริง ควรตั้งค่า TURN_URL และ Credential ใน env เพื่อให้วิดีโอคอลไม่หลุดเมื่อเจอเน็ตที่บล็อก P2P

HTTPS & Local Testing: เบราว์เซอร์จะไม่อนุญาตให้ใช้กล้องและไมค์หากไม่ใช่ HTTPS (Secure Context)
